01 — Problem Statement

Every afternoon shift
starts blind.

Production supervisors inherit a kitchen they didn't open. The information to run a great shift already exists — it just isn't surfaced at the right time, for the right person.

01
Reactive Stockouts

Products run out because prep decisions are made without real-time sales velocity data. By the time the gap is visible, it's too late to bake.

02
Excess Waste

Over-preparation happens because the supervisor doesn't know what was already prepped in the morning. Unsold product at close is a direct cost.

03
Slow Decisions

The first 15–30 minutes of every shift are spent gathering context — time that should be spent managing production at the highest-traffic window of the day.

"The information to run a great shift already exists. It just isn't surfaced at the right time, in the right place, for the right person."

FinTech Parallel

The shift gap mirrors a common fintech problem: a portfolio manager inheriting a position with no context on why trades were made. ShiftSense applies the same design pattern — real-time data surfacing, intelligent alerts, and automated reporting — to an operational context. The architecture is directly transferable.

02 — User Personas

Two users.
One source of truth.

ShiftSense serves two distinct user types with different relationships to the data. The supervisor lives in it. The manager reads from it.

Primary User
The Production Supervisor
"I want to know what's happening in the store before I even put on my apron."
Shift Afternoon / Bake (2:00 PM – 10:00 PM)
Core Need Context on arrival + proactive alerts + frictionless handoff
Frustration Making prep decisions blind. Discovering stockouts after a customer notices.
Success Full shift picture in under 2 minutes on login.
Secondary User
The Store Manager / GM
"I want to see how each shift performed without having to ask anyone."
Frequency Daily — reviews shift reports and trend data
Core Need Consolidated shift performance data without manual aggregation
Frustration Inconsistent verbal reports that aren't searchable or comparable.
Success Opens a shift report and immediately sees what sold, what wasted, what went wrong.

06 — Risks & Mitigations

What could go
wrong.

Risk Impact Likelihood Mitigation
Manual data fatigue Supervisors stop logging if entry feels like extra work on a busy shift High Minimize input fields. Default values where possible. Mobile-optimized forms with one-tap logging.
Threshold misconfiguration Wrong low-stock thresholds cause alert noise or missed stockouts Medium Provide sensible defaults per product category. Allow easy in-app threshold editing per shift.
Shift report ignored Manager doesn't read reports, reducing accountability loop Medium Keep reports scannable — one screen, visual summary first. Add push notification option in Phase 2.
Data accuracy drift Manual inventory counts diverge from reality over time Medium Add a daily inventory reset prompt at shift start to re-anchor all counts.
Single-store bias Patterns learned don't generalize to other locations Low V1 is explicitly single-store. Multi-location is a Phase 2 research question after core value is proven.
07 — Decisions Log

Why we said
no to things.

Every significant scope decision is documented with its rationale. What you leave out is as important as what you include.

Manual data entry in V1

POS integration requires vendor access, API contracts, and engineering overhead that would delay launch by months. The core hypothesis — that surfacing operational data changes supervisor behaviour — can be tested with manual entry. Integration is the right Phase 2 investment once the value is validated, not before.

Web-first, tablet-optimised (no native app)

Kitchen supervisors work on shared tablets, not personal phones. A web app eliminates device dependency, reduces release overhead, and fully covers the use case. Native app development is not warranted until usage patterns justify the investment.

No customer data in V1

The problem is internal and operational. Customer data introduces privacy obligations, compliance requirements, and scope complexity that are not needed to solve the shift gap. This boundary is intentional — not a limitation.

Single store scope for V1

Multi-location rollout requires store comparison logic, role management, and data isolation that add significant complexity before the core product is proven. One store allows fast iteration on real feedback and a clean foundation to scale from.

08 — Next Steps

What happens
next.

01
Validate user stories

Share this PRD with one additional production supervisor. Confirm pain points, story priorities, and any missing scenarios before Phase 1 scope is locked.

02
Resolve open questions

Answer threshold logic, product list scope, and shift ownership questions before the data model is finalized. These decisions affect schema design directly.

03
Build low-fidelity wireframes

Sketch the shift snapshot, inventory view, alert panel, and waste logger. Validate with the primary user before any code is written.

04
Finalize Phase 1 data model

Confirm entity relationships, default threshold values, and shift report schema. Document the final model as a shared reference for the engineering lead.

05
Ship a working prototype

Target a functional Phase 1 prototype within 3–4 weeks for a live shift test with the primary user. Real usage data will drive Phase 2 prioritization.
